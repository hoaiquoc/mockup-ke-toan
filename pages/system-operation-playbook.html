Playbook Vận Hành Hệ Thống Định Khoản

Trang này minh họa cách dùng hệ thống xác định bút toán theo Rule + Mapping + Resolver để tránh hard-code và dễ migration hệ thống mới.

Bắt đầu trong 5 phút (cho cả Dev và Kế toán mới)

Nếu bạn là Dev chưa biết kế toán

  1. Không cần nhớ 131/331/511 ngay, chỉ cần hiểu RoleKey.
  2. Luồng chuẩn: Event -> Rule -> Resolve Role -> Journal.
  3. Nếu thiếu mapping, fail fast và báo lỗi cấu hình, không đoán GUID.

Nếu bạn là Kế toán mới

  1. Chọn đúng loại chứng từ và chi nhánh.
  2. Kiểm tra phần "Ví dụ đời thường" trong từng tab.
  3. Đối chiếu nhanh bằng Sổ nhật ký và Sổ chi tiết sau khi ghi nhận.

Sơ đồ vận hành 6 bước

1. Chuẩn hóa Event

Chứng từ được đổi thành event chuẩn.

2. Chọn Rule

Tìm rule theo event + điều kiện + ngày hiệu lực.

3. Resolve Account

RoleKey -> AccountId theo branch/company/global.

4. Tính số tiền

Áp công thức VAT, giá vốn, phân bổ.

5. Validate cân đối

Tổng debit = tổng credit, role không thiếu.

6. Ghi Journal

Ghi sổ trong transaction + lưu trace version.

Playbook theo đúng 17 chức năng

Mỗi chức năng là 1 tab riêng, có mapping role cụ thể để resolve account (131/331/511/632/3331/133...) mà không hard-code GUID.

Checklist go-live không cần rà soát seed data

  1. Chuẩn hóa RoleKey và EventType chuẩn, không gắn GUID vào code.
  2. Import rule/mapping bằng file config versioned (json/yaml).
  3. Chạy dry-run replay 1 kỳ dữ liệu và so sánh trial balance.
  4. Bật cảnh báo nếu role thiếu mapping tại branch đang xử lý.
  5. Lưu trace RuleVersion + MappingVersion trên mỗi journal entry.

    Flow lên báo cáo từ cấu hình hạch toán

    Nguyên tắc quan trọng: Rule/Mapping chỉ dùng lúc ghi sổ. Khi lên báo cáo, hệ thống đọc từ journal_lines đã hạch toán, không chạy lại rule.

    1. Flow lên Sổ nhật ký chung

    1. Lấy journal entry + journal line đã ghi sổ.
    2. Join sang accounts để hiển thị mã/tên tài khoản.
    3. Nếu xem chi nhánh: filter theo branch_id.
    4. Nếu xem toàn công ty: lấy tất cả branch thuộc công ty.
    5. Sort theo ngày hạch toán, số chứng từ, số dòng.

    2. Flow lên Sổ chi tiết tài khoản

    1. Chọn tài khoản hoặc nhóm tài khoản.
    2. Tính số dư đầu kỳ trước fromDate.
    3. Lấy phát sinh trong kỳ.
    4. Tính số dư chạy theo từng dòng.
    5. Cho drill-down về chứng từ gốc.

    3. Cơ chế xem CN / Toàn công ty

    1. Xem CN: filter đúng 1 branch_id.
    2. Xem Toàn công ty: lấy tất cả branch được phép xem.
    3. Chi tiết: giữ nguyên từng dòng, thêm cột chi nhánh.
    4. Tổng hợp: group theo account_id, parent_id hoặc canonical_account_key.
    5. Không group bằng text account code thuần túy.

    Pseudo-code C# - Query Sổ nhật ký chung

    public async Task> GetJournalReportAsync(
    DateTime fromDate,
    DateTime toDate,
    string scope,
    string? branchId,
    CancellationToken ct)
{
    var branchIds = await ResolveBranchScopeAsync(scope, branchId, ct);

    return await _db.JournalLines
        .AsNoTracking()
        .Where(x => x.PostingDate >= fromDate && x.PostingDate <= toDate)
        .Where(x => branchIds.Contains(x.BranchId))
        .OrderBy(x => x.PostingDate)
        .ThenBy(x => x.JournalEntry.SourceVoucherNo)
        .ThenBy(x => x.LineNo)
        .Select(x => new JournalReportRowDto(
            x.PostingDate,
            x.BranchId,
            x.JournalEntry.SourceVoucherNo,
            x.AccountId,
            x.Account.Code,
            x.Account.Name,
            x.Debit,
            x.Credit,
            x.Description
        ))
        .ToListAsync(ct);
}

    Pseudo-code C# - Query Sổ chi tiết tài khoản

    public async Task GetAccountLedgerAsync(
    DateTime fromDate,
    DateTime toDate,
    string scope,
    string? branchId,
    Guid? accountId,
    string? canonicalAccountKey,
    CancellationToken ct)
{
    var branchIds = await ResolveBranchScopeAsync(scope, branchId, ct);

    var query = _db.JournalLines
        .AsNoTracking()
        .Where(x => branchIds.Contains(x.BranchId));

    if (accountId.HasValue)
        query = query.Where(x => x.AccountId == accountId.Value);
    else if (!string.IsNullOrWhiteSpace(canonicalAccountKey))
        query = query.Where(x => x.Account.CanonicalAccountKey == canonicalAccountKey);

    var opening = await query
        .Where(x => x.PostingDate < fromDate)
        .SumAsync(x => x.Debit - x.Credit, ct);

    var rows = await query
        .Where(x => x.PostingDate >= fromDate && x.PostingDate <= toDate)
        .OrderBy(x => x.PostingDate)
        .ThenBy(x => x.LineNo)
        .Select(x => new LedgerRowDto(
            x.PostingDate,
            x.BranchId,
            x.JournalEntry.SourceVoucherNo,
            x.Debit,
            x.Credit,
            x.Description
        ))
        .ToListAsync(ct);

    decimal running = opening;
    foreach (var row in rows)
    {
        running += row.Debit - row.Credit;
        row.RunningBalance = running;
    }

    return new AccountLedgerReportDto(opening, rows, running);
}

    Nguyên tắc mapping để xem Toàn công ty

    • Nếu toàn công ty dùng chung một tài khoản thì group theo account_id.
    • Nếu mỗi chi nhánh có tài khoản con riêng thì group theo parent_id hoặc canonical_account_key.
    • account_id là khóa bền cho dữ liệu lịch sử; account code chỉ là lớp hiển thị.
    • Nhật ký chung toàn công ty là lấy toàn bộ dòng journal của các chi nhánh, không phải chạy lại rule.
